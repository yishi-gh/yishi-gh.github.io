1 数据库基本概念

数据库是有组织，可共享，统一管理的数据集合。数据库解决了以下问题：

  • 减少数据重复存储的冗余
  • 利用事务机制，确保操作对不同对象的一致性
  • 允许多个用户同时访问操作数据而不会互相覆盖
  • 提供精细的权限控制
  • 使用索引技术提供高效的检索

数据库包含以下三个层次：

  • 数据库（Database）：实际上存储在硬盘上的数据
  • 数据库管理系统（DBMS）：操作和管理数据库的软件
  • 数据库系统（DBS）：一个完整的运行环境，包括数据库、DBMS、硬件、软件以及管理员

根据数据的逻辑关系，又可以分为关系型/非关系型
关系型例如

  • MySql（基于表，插件式存储引擎-应用于中小规模开发）
  • Oracle（基于表-高性能高维护成本）
    优点是格式一致，易于维护，SQL语言通用，支持复杂操作。缺点是读写性能较差，数据结构灵活度受限，高并发瓶颈

非关系型例如

  • Redis（基于键值对-应用于缓存）
  • MongoDB（基于bson文档，类似json-应用于高性能低事务场景）
    优点是数据结构灵活，易于扩展，高并发，缺点是缺乏统一标准，语言不通，迁移成本高，事务支持弱，数据存储冗余

2 数据库构成

2.1 表 Table

表是用于组织和存储数据的矩阵结构，包含一组字段（Columns）和记录（Rows）。一个表对应E-R模型中的一个实体集，一行对应一个实体，也叫一条记录，一个列对应一个属性，也叫一个字段。

2.2 关系 Relationship

关系通过公共字段连接不同的表，可以是一对一、一对多、多对多或自我引用。

2.2.1 一对一

一对一应用不多，因为可以创建为一张表，举例来说，比如将一张学生信息表拆分为两个表，二者通过 学号 相联系。
两种建表原则：

  • 外键唯一：主表的主键和从表的外键形成主外键关系，且外键唯一
  • 外键是主键：主表的主键和从表的主键形成主外键关系。
    两个原则的目的都是确保主表的一条记录只能对应从表的一条记录。外键唯一原则下，主从表的主键可以各自增长。外键是主键原则下，从表没有独立的ID，只是从主表中借来，完全依附于主表

2.2.2 一对多

在从表创建一个字段，字段作为外键指向主表的主键

2.2.3 多对多

创建第三个表（联接表），将多对多关系拆分为两个一对多关系，将这两个表的主键插入到第三个表中。
举例：学生表+开课表->选课表，选课表中同时包含学生主键和课程信息主键，一个学生可以选多门课，一门课也可以有多个学生

2.2.4 自我引用

举例：如员工表中，某些员工的主管可能也出现在这张表中，其主管编号同时也是员工编号

2.3 索引 Index

基于一个或多个列的值创建，用于提高数据的检索排序效率

2.4 查询语言 SQL

用于与数据库交互

2.5 字符集编码

2.5.1 字符集简介

字符集用于描述某个字符范围的编码规则，解决了界定字符范围和映射规则的问题
常用字符集有：

  • ASCII 字符集
    共收录128字符，包括空格、标点、数字、大小写字母、不可见字符。采用1字节编码
  • ISO 8859-1 字符集
    共收录256字符，在ASCII基础上又扩充西欧常用字符，采用1字节编码
  • GB2312 字符集
    收录汉字、拉丁字母、希腊字母、日文、俄语，兼容ASCII，若字符在ASCII中则采用1字节编码，否则采用2字节编码。在这种变长编码方式中，若某字节在0-127内，则单字节编码，否则双字节编码。
  • GBK 字符集
    扩充GB2312，编码方式兼容GB2312
  • Unicode 字符集
    收录地球上几乎所有字符，兼容ASCII，采用变长编码，一个字符需要1-4个字节。Unicode可以采用UTF-8，UTF-16，UTF-32等编码。MySQL不区分字符集和编码方案的概念

UTF-8 需要1-4个字节编码一个字符，但常用的字符只需要1-3个字节，故MySQL中默认utf8mb3为UTF8，采用1-3个字节编码，而真正的utf8则是utf8mb4

3 数据类型

3.1 整数

类型 占用空间 (字节) 有符号范围 (Signed) 无符号范围 (Unsigned) 适用场景建议
TINYINT 1 Byte -128 ~ 127 0 ~ 255 状态值、布尔模拟、小选项
SMALLINT 2 Bytes -32,768 ~ 32,767 0 ~ 65,535 班级人数、年份、小范围ID
MEDIUMINT 3 Bytes -8,388,608 ~ 8,388,607 0 ~ 16,777,215 中型系统流水、城市人口
INT 4 Bytes -2,147,483,648 ~ 2,147,483,647 0 ~ 4,294,967,295 常规主键 ID、用户量
BIGINT 8 Bytes -9.22e18 ~ 9.22e18 0 ~ 1.84e19 全球级 ID、时间戳、财务总额

4 类型约束

4.1字段约束

  • unsigned 表示无符号
  • 字符类型后加括号表示限制宽度。int默认无符号为11，有符号为10，只有在给int设置前导填零时，宽度才有意义，如 int(4) unsigned zerofill
  • not null 表示不能为空，若输入NULL则报错
  • default设置默认值
  • primary key主键唯一且不能为空
  • auto_increment定义列为自增，一般用于主键
  • unique为唯一索引，数据不能重复

4.2 主键

主键是用于唯一标识每一行的一列，任意两行的主键值都不相同。主键可以不用但不能没有，方便日后扩展。
要求：

  • 主键最好不要修改
  • 不允许NULL
  • 主键在逻辑上应该不变
  • 主键是唯一的，但可以使用多列作为主键，称为联合主键。在这种情况下，所有列值的组合必须是唯一的。

6 存储引擎

在MySQL中，数据的存储和提取都被封装到存储引擎中。存储引擎负责对于逻辑操作的物理实现、具体如何读取数据写入存储器。不同的引擎的存储结构不尽相同，存取算法也不尽相同。

  • InnoDB
    默认引擎，最常用。
    支持事务、行级锁、支持外键、聚簇索引。
    适用于需要事务支持，高并发的场景
