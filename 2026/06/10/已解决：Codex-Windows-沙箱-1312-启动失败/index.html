解决

公开 issue 中有一个非常接近的案例：Windows 下 Codex shell command 失败，报 CreateProcessAsUserW failed: 1312，并且环境里使用的是 WindowsApps / Store 版 pwsh.exe。

这个 issue 给出的关键启发是：问题可能不在 Codex 权限配置，而在 WindowsApps / MSIX 版 PowerShell 的启动方式。Codex 沙箱需要在受限 token 下启动 shell，WindowsApps 的 App Execution Alias / Store activation 路径在这种上下文里可能失败。

之前由于 PowerShell 5.1 默认 GBK 编码，可能导致终端输出中文乱码。笔者根据微软文档的推荐使用 winget 安装了 PowerShell 7。

关键在于 PowerShell 7.6 起通过 winget 默认安装的是 MSIX 包。这种包的好处在于支持自动更新。但微软在下方注解中提到，MSIX 只会为单个用户安装，且本身在沙盒中运行。这可能与 Codex 使用受限 token 启动 shell 子进程相冲突。

但笔者没有足够的能力进一步探究原因，本文只讨论解决方案。下面有由 Codex 生成的原因分析，仅供参考。

分析

分析部分完全由 Codex 生成，仅供参考。

这个问题的关键不在 Codex 的 workspace-write 配置，而在 Windows shell 启动链路。

问题链路可以概括为：

1312 对应 Windows 错误 ERROR_NO_SUCH_LOGON_SESSION，含义是指定登录会话不存在或已经终止。这里的失败发生在创建进程阶段，而不是 PowerShell 脚本执行阶段。

MSIX/WindowsApps 版和 MSI/WiX 版不只是路径不同，还包括安装模型、启动方式、隔离模型和更新机制的差异。MSIX 更适合普通用户安装和自动更新；MSI/WiX 更适合 CLI 工具、IDE、CI、agent、服务和沙箱子进程调用。

本次修复没有放宽到整机权限，也没有切换到 danger-full-access。仍然保留：
